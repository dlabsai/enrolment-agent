# Evals

## Context
The Evals page lets admins review evaluation reports generated by the CLI LLM
judge tests. Reports are stored as Markdown files in the backend `reports/`
directory and expose the same summary tables and failure details produced by
`app.evals`.

## Data Sources
- `GET /evals/reports`
  - Auth: Admin/Dev only.
  - Returns metadata for available reports (id, name, generated_at, repeats,
    concurrency, filename, size_bytes).
  - Sorted by `generated_at` descending.

- `GET /evals/reports/{report_id}`
  - Auth: Admin/Dev only.
  - Returns report metadata plus full Markdown content.

- `GET /evals/reports/{report_id}/log`
  - Auth: Admin/Dev only.
  - Returns the persisted log file for the selected report.

- `GET /evals/test-cases?suite=...`
  - Auth: Admin/Dev only.
  - Returns test case IDs for the selected suite.

- `POST /evals/runs/stream`
  - Auth: Admin/Dev only.
  - Body: `suite` (chatbot|guardrails|search), `repeat`, `max_concurrency`,
    `pass_threshold`, `test_cases` (optional comma-separated IDs),
    `chatbot_model`, `search_model`, `guardrail_model`, `extractor_model`,
    `evaluation_model` (optional overrides).
  - Streams server-sent events with log output and completion status.
  - Emits a `report` event when a new report file is written.
  - The `start` status event includes a `log_id` to fetch the stored output.

- `GET /evals/runs/logs/{log_id}`
  - Auth: Admin/Dev only.
  - Returns log file metadata plus full text content.

## UI Structure
- Header
  - Title: "Evals".
  - View toggle: Report / Compare / Trends / Models.
  - Refresh action.

- Run evals card
  - Suite selector (Chatbot, Guardrails, Search).
  - Inputs for repeats, concurrency, pass threshold, and optional test case IDs
    via a searchable multi-select.
  - Model selection dialog for Chatbot, Search, Guardrails, Extractor, and Judge
    models, with preset save/delete.
  - Run/Stop actions and a live log stream.
  - Button to expand the terminal.
  - Status badge reflecting running/complete/error states.

- Two-column layout
  - Reports list card
    - Search input (client-side filter).
    - Table columns: report name + filename, generated time, repeats,
      concurrency.
  - Detail card (Report view)
    - Shows selected report name and metadata (generated, repeats, concurrency,
      size).
    - View log action to open the persisted run log for the report.
    - Markdown viewer for the full report contents (summary table, averages,
      failed runs).
  - Detail card (Compare view)
    - Selectors for report type, baseline run, compare run, and a swap action.
    - Metadata for both runs (generated time, repeats, concurrency, size).
    - Model configuration comparison table (role → baseline vs compare).
    - Comparison table with per-case pass rate, runtime error rate, median
      duration, and deltas.
  - Detail card (Trends view)
    - Selector for report type and multi-select for reports in the charts.
    - Trend charts for pass rate and median duration, with model names shown in
      tooltips.
  - Detail card (Models view)
    - Selector for report type.
    - Model groups table keyed by chatbot/guardrails/search combinations with
      median pass rate and median median duration.
    - Reports list for the selected model group with per-report metrics.

## Behavior Notes
- Default selection is the most recent report.
- Compare mode defaults to the latest report type and its two most recent runs.
- Compare mode is scoped to report types (same `name`) and requires at least two
  runs to render deltas.
- Trends mode defaults to the latest report type and selects the most recent
  five runs for charting.
- Trend charts plot average pass rate plus average median duration derived from
  the summary table.
- Models mode defaults to the latest report type and groups reports by the
  chatbot/guardrails/search model combination.
- Model groups show median pass rate and median median duration across reports
  in the combo, with a per-report breakdown list. Non-applicable model roles are
  omitted for search/guardrails-only report types.
- Running evals streams live log output while the backend executes pytest and
  writes the same output to a log file for on-demand viewing. The first log line
  echoes the full pytest command.
- Report detail view can open the stored log for past runs.
- The terminal output does not cap log line count in the UI.
- Test case IDs are fetched per suite and can be selected or cleared without
  manual comma entry.
- Successful runs refresh the report list and select the newly generated report.
- Model overrides and presets persist in local storage and apply to subsequent
  runs.
- Refresh reloads the report list and clears cached details.
- Empty states:
  - List shows "No reports" message when none exist.
  - Detail panel prompts selection when no report is available.
  - Compare panel explains when fewer than two runs exist for a type.
- Errors surface retry actions without crashing the page.
